# PR Review Skill Design

**Date:** 2026-04-23
**Issue:** [#5](https://github.com/eakleven/cyfor-ai-workshop/issues/5)

## Goal

Create a minimal, reusable skill at `.claude/skills/review-pr/SKILL.md` that guides the model through a consistent PR review for this repository.

## Approach

Pure checklist (Option A). No rationale prose, no output template — just the checks that matter for this project, grouped for readability.

## Skill location

`.claude/skills/review-pr/SKILL.md`

## Checklist structure

### 1. Fetch the PR
- Run `gh pr view <number> --comments` to see description, labels, and existing comments.
- Run `gh pr diff <number>` to see the full diff.

### 2. API ↔ Web sync
- If `api/src/app.ts` or any Zod schema changed → verify `api/openapi.json` was updated.
- If `api/openapi.json` changed → verify `web/src/api/generated/hooks.ts` was regenerated by Orval.
- `web/src/api/generated/**` must never be hand-edited (Orval has `clean: true`).

### 3. Generated files
- If `api/prisma/schema.prisma` changed → Prisma client must be regenerated.
- `npm run generate` must have been run after any schema or route change.
- No generated file should have manual edits on top of generated output.

### 4. Domain correctness
- Zod schemas align with the resource/booking domain (sensible field names, types, required/optional split).
- Validation is enforced at API boundaries (route input schemas), not only in the frontend.
- No validation gaps: missing-record cases, bad enum values, empty strings treated as valid.

### 5. Code quality
- Change is small and focused — one concern per PR.
- No obvious bugs.
- Edge cases handled (resource not found, invalid input, concurrent updates).
- No risky assumptions that could silently break in production.

### 6. Verdict
- Approve, request changes, or flag blockers with specific file:line references.

## Out of scope

- Style/formatting preferences.
- Unrelated refactoring.
- Performance micro-optimisations unless they introduce a risk.
